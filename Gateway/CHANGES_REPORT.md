# Gateway — повторный анализ и исправления

## Что было не так

### 1. Конфигурация Spring Security
- `SecurityConfig` открывал весь `/auth/**`, поэтому любой новый endpoint внутри `/auth` автоматически становился публичным.
- Не было нормального разделения поведения для браузерных страниц и REST-запросов: один и тот же `AuthenticationEntryPoint` всегда пытался редиректить пользователя.
- Из-за этого API-клиенты могли получать `302` вместо ожидаемого `401`.

### 2. AuthenticationEntryPoint
- Был редирект на `/login`, хотя реальная страница логина находится по `/auth/login`.
- Для браузерных запросов была идея авто-refresh через `refresh_token`, но она использовала небезопасный `redirect`-параметр дальше по цепочке.
- Для не-HTML запросов не было нормального `401`-ответа.

### 3. JWT фильтр
- На каждый защищённый запрос фильтр ходил во внешний `user-service` через `CustomUserDetailsService.loadUserById(...)`.
- Это делало gateway зависимым от доступности внешнего сервиса даже для обычной проверки access token.
- В фильтре в principal складывался `UserDetails` с паролем, хотя пароль там не нужен.
- При ошибках разбора токена контекст явно не очищался.

### 4. BlackList / Redis
- Если записи о пользователе не было в Redis, токен считался невалидным.
- После очистки Redis все токены внезапно переставали работать.

### 5. Контроллеры
- `GET /auth/refresh` принимал redirect как есть и мог сделать редирект на внешний сайт (`open redirect`).
- Logout менял состояние сервера через `GET /logout`.
- Logout инвалидировал токены только если удавалось достать access token.
- Повторялась логика работы с cookie в нескольких местах.

### 6. Конфигурация приложения
- `access-token-expiration: 10000` не соответствовал комментарию `15 минут`; по факту это было 10 секунд.
- Работа с `Secure` cookie не учитывала сценарий reverse proxy / TLS termination.

---

## Что я изменил

### Новые классы
#### `ru.mescat.security.util.AuthCookieService`
Вынес в отдельный сервис:
- создание auth-cookie,
- очистку cookie,
- извлечение cookie,
- выбор токена из cookie или `Authorization` header,
- определение secure-соединения с учётом `X-Forwarded-Proto`.

Зачем:
- убрать дублирование,
- сделать поведение login / refresh / logout единым,
- корректнее работать за reverse proxy.

#### `ru.mescat.security.JwtPrincipal`
Добавил лёгкий principal только с `UUID userId`.

Зачем:
- не тащить пароль в `SecurityContext`,
- не грузить `UserDetails` во время проверки access token.

---

## Изменённые файлы

### `SecurityConfig.java`
Что изменено:
- сузил `permitAll()` до конкретных endpoints,
- добавил `POST /auth/logout` в публичные endpoints,
- оставил stateless security,
- оставил JWT-фильтр перед `UsernamePasswordAuthenticationFilter`.

Зачем:
- убрать слишком широкий доступ к `/auth/**`,
- оставить публичными только реально публичные точки входа.

### `CustomAuthenticationEntryPoint.java`
Что изменено:
- для HTML-запросов:
  - если есть `refresh_token` → redirect на `/auth/refresh?redirect=...`,
  - если refresh нет → redirect на `/auth/login`.
- для API / non-HTML запросов теперь отдаётся обычный `401` и текст `Требуется авторизация`.

Зачем:
- браузеру нужен redirect-flow,
- API нужен нормальный статус-код, а не HTML-редирект.

### `JwtAuthenticationFilter.java`
Что изменено:
- убран вызов `userDetailsService.loadUserById(...)` на каждый запрос,
- access token теперь проверяется локально:
  - подпись,
  - тип `access`,
  - срок жизни,
  - blocklist,
- в `SecurityContext` кладётся `JwtPrincipal`,
- при любой ошибке контекст очищается.

Зачем:
- снизить связанность с user-service,
- убрать лишнюю передачу пароля,
- сделать access-flow быстрым и предсказуемым.

### `BlackListTokens.java`
Что изменено:
- если записи в Redis нет, токен теперь считается **валидным**, а не невалидным.

Зачем:
- не ломать все токены после потери Redis-состояния,
- сделать blocklist устойчивее к очистке кэша.

### `CustomUserDetailsService.java`
Что изменено:
- оставлен только сценарий `loadUserByUsername(...)` для логина по паролю,
- удалён `loadUserById(...)`, который раньше тянул пароль в JWT-flow.

Зачем:
- пароль нужен только в password-authentication,
- в JWT-фильтре он не нужен.

### `AuthController.java`
Что изменено:
- вынесена работа с cookie в `AuthCookieService`,
- `GET /auth/refresh` теперь санитизирует redirect,
- при неудачном browser-refresh куки очищаются и пользователь возвращается на `/auth/login`.

Зачем:
- убрать open redirect,
- не оставлять клиенту протухшие токены после неуспешного refresh.

### `LogoutController.java`
Что изменено:
- logout переведён на `POST /auth/logout`,
- при logout ищется любой доступный токен: access, refresh или Bearer,
- если подпись валидна — инвалидируется блоклистом весь выпуск токенов пользователя,
- затем cookie очищаются.

Зачем:
- перестать менять состояние на `GET`,
- корректно завершать сессию даже когда access token уже протух, а refresh ещё есть.

### `RefreshService.java`
Что изменено:
- оставлена полная проверка refresh token,
- добавлена проверка `user.isBlocked()`,
- перед выдачей новых токенов гарантируется инициализация записи в blocklist.

Зачем:
- не выпускать токены для заблокированного пользователя,
- не зависеть от случайного отсутствия записи в Redis.

### `application.yaml`
Что изменено:
- `access-token-expiration` исправлен на `900000` (15 минут),
- добавлен `server.forward-headers-strategy: framework`.

Зачем:
- привести срок жизни access token к ожидаемому,
- корректно определять HTTPS за reverse proxy.

### `templates/index.html`
Что изменено:
- исправлена подсказка с неправильных путей,
- оставлена рабочая навигация на `/auth/login` и `/auth/reg`.

---

## Что осталось осознанно простым
- Я не стал вводить роли и `GrantedAuthority`, потому что в текущем проекте они нигде не используются.
- Я не менял контракт внешнего `user-service`; gateway по-прежнему использует его для login / register / refresh-проверок пользователя.
- Я не выносил JWT secret в env-переменные автоматически, потому что это уже зависит от окружения запуска, но для production это обязательно стоит сделать.

---

## Итого
Исправленная версия теперь:
- не редиректит API-клиентов вместо `401`,
- не использует пароль в JWT-фильтре,
- не ходит во внешний user-service на каждый защищённый запрос,
- не уязвима к open redirect через `/auth/refresh`,
- корректнее работает с logout,
- не ломает все токены при пустом Redis,
- использует реальный TTL access token в 15 минут.
